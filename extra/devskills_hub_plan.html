Autenticazione & Autorizzazione — JWT, OAuth2, RBAC
Giorni 21–35 · 15 giorni Studio + Build
▾
// Concetti chiave che imparerai
JWT (JSON Web Token): un token firmato crittograficamente che il client conserva (localStorage o cookie HttpOnly) e invia ad ogni richiesta. Il server lo verifica senza consultare il DB — è stateless. Struttura: Header.Payload.Signature.

OAuth2: un protocollo di delegazione: "permetti a Google di dire che sei chi dici di essere". Non è un'alternativa a JWT — spesso si usano insieme: OAuth2 gestisce il flusso di login sociale, JWT viene poi emesso dal tuo backend come token di sessione.

RBAC (Role-Based Access Control): ogni utente ha un ruolo, ogni ruolo ha dei permessi. Nel nostro caso: student può leggere corsi e iscriversi; instructor può creare e modificare i propri corsi; admin può fare tutto.
giorno21
JWT — Teoria Profonda
  • Studia (2h): anatomia di un JWT (header, payload, signature), come viene firmato con HMAC-SHA256 o RS256, perché non va messa informazione sensibile nel payload (è base64, non cifrato)
  • Differenza tra Access Token (breve durata, 15min) e Refresh Token (lunga durata, 7 giorni)
  • Rischi: token theft, come mitigarli (HttpOnly cookie per refresh token, HTTPS obbligatorio)
  • Prova a decodificare un JWT su jwt.io per capirne la struttura
giorno22
JWT — Implementazione con SimpleJWT
  • Installa djangorestframework-simplejwt e configura in settings.py
  • Endpoint POST /api/auth/token/ → riceve email+password, restituisce access+refresh token
  • Endpoint POST /api/auth/token/refresh/ → scambia il refresh token per un nuovo access token
  • Configura il refresh token come cookie HttpOnly lato server (più sicuro di localStorage)
  • Testa il ciclo completo: login → chiamata API protetta → scadenza → refresh → nuova chiamata
giorno23
OAuth2 — Teoria: il Flusso Authorization Code
  • Studia (2h): i 4 ruoli OAuth2 (Resource Owner, Client, Authorization Server, Resource Server), i grant types (Authorization Code, il più sicuro per web app)
  • Traccia il flusso passo per passo: utente clicca "Login con Google" → redirect a Google → Google chiede consenso → redirect al tuo backend con un code → il backend scambia il code con un access_token Google → il backend verifica l'identità → emette il proprio JWT
  • Registra una app su Google Cloud Console e GitHub Developer Settings per ottenere client_id e client_secret

  • Endpoint POST /api/auth/google/: riceve il token Google dal frontend e restituisce il JWT della tua app

  • Studia (1h): differenza tra RBAC (ruoli predefiniti) e ABAC (attributi dinamici) — per il nostro progetto RBAC è sufficiente
  • Crea 3 ruoli: student, instructor, admin. Ogni ruolo mappa a un set di permessi DRF
  • Scrivi custom permission classes DRF: IsInstructor, IsStudentEnrolled, IsOwnerOrAdmin

💡 Regola d'oro del RBAC: scrivi sempre la permission in modo "positivo" — chi può fare questa cosa? — invece di "chi non può". Poi combina con IsAuthenticated come base.

// PWA — Cosa significa davvero
Una PWA (Progressive Web App) è una web app che si comporta come un'app nativa: installabile sullo schermo del telefono, funziona (parzialmente) offline, riceve notifiche push, ha una splash screen. Si ottiene aggiungendo due cose al tuo React app: un Web App Manifest (JSON che descrive l'app: nome, icona, colori) e un Service Worker (script JS che gira in background, intercetta le richieste di rete e gestisce la cache offline). Con Vite + vite-plugin-pwa si configura in poche ore.

Redis — Caching, Rate Limiting & Sessioni
Giorni 57–66 · 10 giorni Implementazione
▾
// Perché Redis è fondamentale
Redis è un database in-memory estremamente veloce (sub-millisecondo). In un'app come la nostra lo usi per: caching (evita query PostgreSQL ripetute per dati che cambiano poco, come la lista dei corsi popolari), rate limiting (contare le richieste per IP/utente), sessioni, e come broker di messaggi per Celery (fase successiva).

  • Connettiti al container Redis con redis-cli e sperimenta i comandi: SET/GET, HSET/HGET, EXPIRE, TTL, INCR
  • Pattern di naming delle chiavi: course:detail:{slug}, user:{id}:session, rate_limit:{ip}:{endpoint}
  • Comprendi il concetto di TTL (Time To Live): i dati scadono automaticamente dopo N secondi

Celery — Background Jobs, Queue, Workers
Giorni 63–75 · 13 giorni Implementazione
▾
// Perché i background jobs sono essenziali
Alcune operazioni non devono bloccare la risposta HTTP: inviare un'email, generare un PDF, processare un video caricato, inviare notifiche. Celery è un task queue: il tuo backend mette i task in una coda (su Redis), i worker Celery li processano in background. Celery Beat è lo scheduler: esegue task periodici (ogni ora, ogni giorno).

  • Installa celery + redis e configura celery.py nel progetto Django
  • Aggiunge il container celery_worker al docker-compose.yml (stesso codice, comando diverso)
  • Scrivi il tuo primo task: @shared_task def send_welcome_email(user_id): ... e triggera da Django

AWS S3 + CloudFront — Storage & CDN
Giorni 82–93 · 12 giorni Cloud/Infra
▾
// S3 e CloudFront spiegati semplici
AWS S3 è uno storage di oggetti (file) infinitamente scalabile. Ci metti dentro video, immagini, PDF, e sono accessibili tramite URL. Non è un file system — non hai cartelle reali, hai bucket con oggetti il cui nome può contenere /.

CloudFront è la CDN (Content Delivery Network) di AWS: distribuisce i tuoi file S3 da server presenti in tutto il mondo. Un utente in Giappone scarica il video dal server AWS più vicino a lui, non dal tuo bucket in us-east-1. Risultato: latenza bassa e costo ridotto (CloudFront è più economico di S3 per le richieste di download).

  • Studia IAM (1h): Users, Groups, Roles, Policies. Principio del minimo privilegio.
  • Crea un utente IAM con policy AmazonS3FullAccess (solo per il bucket del progetto), genera le access keys
  • Crea due bucket S3: devskills-media (video, immagini) e devskills-private (certificati PDF — accesso solo tramite signed URL)
giorno83-84
Django → S3 con boto3 (2 giorni)
  • Installa boto3 e django-storages. Configura come default storage backend
  • I file caricati via Django (thumbnail corsi, avatar) vanno automaticamente su S3
  • Upload diretto dal frontend a S3 con Presigned URL: il backend genera un URL temporaneo firmato, il client carica il video direttamente su S3 — il file non passa per il tuo server (molto più scalabile)
  • Testa l'upload di un video da Postman usando il presigned URL
💡 Presigned URL: è la tecnica corretta per upload di file grandi. Endpoint: POST /api/lessons/{id}/upload-url/ → restituisce URL firmato S3 valido 10 minuti → il client fa PUT direttamente su S3.

DevOps — Docker Prod, CI/CD GitLab, AWS ECS & Load Balancer
Giorni 89–114 · 26 giorni DevOps/Infra
▾
// CI/CD e Load Balancer spiegati
CI/CD (Continuous Integration / Continuous Deployment): ogni volta che fai push su Git, una pipeline automatica esegue i test, costruisce l'immagine Docker e la deploya in produzione — senza intervento manuale. GitLab CI è gratuito per progetti pubblici/privati (2000 minuti/mese su GitLab.com) e si configura con un file .gitlab-ci.yml.

AWS Application Load Balancer: distribuisce il traffico HTTP/HTTPS in arrivo verso più istanze della tua app (i container ECS). Gestisce SSL termination (il certificato HTTPS sta qui, non nei container), health checks (toglie dal pool le istanze non sane), e path-based routing (es. /api/* → backend, /* → frontend statico su S3).
giorno89-91
Docker Production — Multi-stage Builds (3 giorni)
  • Studia (1h): differenza tra Dockerfile di sviluppo (volumi, hot reload) e produzione (immagine ottimizzata, senza devDependencies)
  • Riscrivi il Dockerfile backend con multi-stage build: stage "builder" che installa dipendenze, stage finale minimale che copia solo il necessario — immagine finale più piccola del 60%
  • Aggiungi Nginx come reverse proxy davanti a Gunicorn (Django): gestisce file statici, compressione gzip, header di sicurezza
  • Testa che il docker-compose di produzione funzioni localmente prima di andare in cloud
giorno92-95
GitLab CI/CD — Pipeline Completa (4 giorni)
  • Crea un progetto su GitLab.com (gratuito, include CI/CD runner condivisi)
  • Scrivi .gitlab-ci.yml con 4 stage: test → build → push → deploy
  • Stage test: esegue pytest con coverage, fallisce se coverage <60%
  • Stage build: costruisce l'immagine Docker
  • Stage push: carica l'immagine su AWS ECR (Elastic Container Registry) con tag del commit SHA
  • Stage deploy: aggiorna il service ECS con la nuova immagine — solo su branch main
💡 GitLab vs Jenkins: per un progetto personale, GitLab CI è la scelta migliore. È completamente gratuito su gitlab.com con 2000 minuti/mese (sufficienti), ha runner già pronti senza configurare server, e la sintassi YAML è intuitiva. Jenkins richiede un server dedicato da mantenere — overhead non giustificato per questo progetto.
giorno96-100
AWS ECS — Deploy in Cloud (5 giorni)
  • Studia (1.5h): ECS Fargate vs EC2 launch type. Fargate è serverless (AWS gestisce i server) — ideale per iniziare perché non devi configurare istanze EC2
  • Crea un ECS Cluster, una Task Definition (definisce i container e le risorse), un ECS Service (mantiene X istanze della task sempre in esecuzione)
  • Configura RDS PostgreSQL managed (non nel container — è più robusto, con backup automatici)
  • Configura ElastiCache Redis managed (stesso ragionamento)
  • Usa AWS Secrets Manager per le variabili d'ambiente sensibili (DB password, Stripe key): i container le leggono al boot senza che siano nel docker-compose

  • Path-based routing: /api/* → backend ECS, /* → bucket S3 con il build React (frontend statico)
