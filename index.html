// I building block di NestJS — impararli nell'ordine giusto
NestJS ha un'architettura fortemente ispirata ad Angular (moduli, dependency injection, decoratori). I concetti vanno imparati in questo ordine:

Module: unità organizzativa. Ogni feature è un modulo (UsersModule, CoursesModule). Il modulo radice è AppModule. I moduli dichiarano cosa esportano e cosa importano da altri moduli.

Controller: gestisce le richieste HTTP. Mappa i path (@Get('/courses')), estrae parametri (@Param, @Body, @Query), delega la logica al Service.

Service: contiene tutta la logica di business. È @Injectable() — NestJS gestisce il suo ciclo di vita e lo inietta dove serve (dependency injection).

DTO (Data Transfer Object): classe TypeScript che descrive la forma dei dati in ingresso. Usata con class-validator per la validazione automatica.

Pipe: trasforma o valida i dati prima che arrivino al controller. La ValidationPipe globale usa i DTO per validare automaticamente ogni richiesta.

Guard: decide se una richiesta può procedere. Esempio: JwtAuthGuard verifica che il token JWT sia valido. Se il guard ritorna false, NestJS risponde automaticamente 401/403.

Interceptor: wrappa la richiesta/risposta. Usi: logging, trasformazione della risposta, caching, timeout globale.

Auth — JWT, OAuth2, Guards & RBAC in NestJS
Giorni 27–40 · 14 giorniStudio + Build

// JWT, OAuth2 e RBAC — la triade dell'autenticazione moderna
JWT (JSON Web Token): token firmato crittograficamente diviso in tre parti (Header.Payload.Signature). Il server lo genera al login, il client lo invia nell'header Authorization: Bearer <token>. Il server lo verifica con la chiave segreta senza query al DB — è stateless. Ha una scadenza (exp): normalmente 15 minuti per l'access token, 7 giorni per il refresh token.

OAuth2: protocollo di delega dell'autenticazione. "Google certifica che questo utente è chi dice di essere." Il tuo backend riceve un token Google, lo verifica, e poi emette il proprio JWT. Non è alternativo a JWT — sono complementari.

RBAC (Role-Based Access Control): ogni utente ha un ruolo (student, instructor, admin), ogni ruolo mappa a permessi. In NestJS si implementa con i Guards — il posto architetturalmente corretto per il controllo accessi.

// Redis, Caching e Load Balancer — il triangolo della scalabilità
Redis è un database in-memory con latenza sub-millisecondo. Nella tua architettura lo usi per: caching delle risposte API frequenti, rate limiting delle richieste, storage dei refresh token (più veloce del DB), e come broker di messaggi per BullMQ (fase successiva).

Load Balancer (teoria): quando hai più istanze del backend (es. 3 container ECS), chi smista le richieste? Il Load Balancer. Distribuisce il traffico in round-robin o per peso, esegue health check e rimuove le istanze non sane. Un problema critico con il caching: se hai 3 istanze e ognuna ha la sua cache in memoria, un utente potrebbe vedere dati diversi a seconda di quale istanza risponde. Redis risolve questo: è una cache condivisa tra tutte le istanze — tutti leggono e scrivono sulla stessa sorgente. Ecco perché Redis e Load Balancer sono strettamente correlati.

Pagamenti — Braintree (Visa/Mastercard) + PayPal
Giorni 98–111 · 14 giorniImplementazione
▾
// Perché Braintree — e come funziona l'architettura di pagamento
Braintree (di PayPal, Inc.) è un gateway di pagamento che gestisce in un unico SDK: Visa, Mastercard, Amex, Discover e PayPal nativo. Questo è esattamente ciò che hai richiesto — Braintree è il wrapper che ti permette di accettare tutte queste metodologie senza costruire integrazioni separate.

Il flusso di sicurezza: (1) Il frontend chiede al backend un client token. (2) Il frontend usa il token per inizializzare il Braintree Drop-in UI — un componente sicuro di Braintree che raccoglie i dati della carta (i dati carta non toccano mai il tuo server). (3) Il Drop-in restituisce un payment method nonce — un token usa e getta. (4) Il frontend invia il nonce al tuo backend. (5) Il backend usa il nonce per creare una transazione su Braintree. Questo schema garantisce la compliance PCI-DSS anche per te.
giorno98-99
Braintree — Setup & Teoria (2 giorni)
  • Crea account Braintree Sandbox (gratuito). Ottieni le 3 credenziali: merchantId, publicKey, privateKey
  • Studia (2h): il flusso Client Token → Drop-in UI → Nonce → Transaction. Differenza tra authorization (riserva i fondi) e settlement (addebito effettivo). Concetto di Vault (salva il metodo di pagamento per pagamenti futuri)
  • Installa braintree Node.js SDK. Configura il gateway nel PaymentsModule
  • Endpoint GET /payments/client-token: genera e restituisce il client token per il frontend

  • Gestisci i casi di errore Braintree: carta rifiutata (processor_declined), CVV errato, fondi insufficienti — mostra messaggi specifici al frontend

AWS S3 + CloudFront — Storage & CDN
Giorni 109–120 · 12 giorniCloud
▾
// S3 e CloudFront — storage e distribuzione globale
AWS S3 è uno storage a oggetti scalabile all'infinito. I file (video, immagini, PDF) vivono in bucket e sono accessibili via URL. Non è un filesystem — non hai cartelle reali, ma il nome dell'oggetto può includere / per simulare una struttura.

CloudFront è la CDN AWS: distribuisce i tuoi file S3 da edge server in tutto il mondo. Un utente in Giappone scarica il video dal punto di presenza AWS più vicino, non dal bucket in us-east-1. Riduce la latenza e il costo (il trasferimento da CloudFront è più economico di S3 diretto).

Presigned URLs: tecnica cruciale — invece di far passare i file caricati per il tuo server (lento, costoso), il backend genera un URL S3 firmato e temporaneo, il client fa upload direttamente a S3. I video non transitano per NestJS.
giorno109-110

  • Studia IAM (1.5h): Users, Groups, Roles, Policies. Come scrivere una policy JSON che permette solo s3:GetObject, s3:PutObject, s3:DeleteObject su un bucket specifico
  • Crea 2 bucket: YouStudent-media (video e immagini, accesso via CloudFront) e YouStudent-private (certificati PDF, accesso solo tramite signed URL)
  • Installa @aws-sdk/client-s3 @aws-sdk/s3-request-presigner nel backend NestJS

  • Frontend Angular: all'upload del video, chiama prima l'endpoint presigned, poi fa PUT direttamente su S3 con progress bar (HttpClient con reportProgress: true)
  • Dopo l'upload: il frontend invia la chiave S3 al backend, che la salva nel record Lesson/Certificate
  • Migra la generazione certificati PDF da filesystem locale a S3
